# Claude AI への開発指示書

## 基本ルール

### 1. コード提供のルール
- **必ず完全なファイル全体を提供すること**
- 部分的な修正箇所のみの提示は絶対に禁止
- 「○○行目付近を修正」のような指示は絶対にしない
- **「主要部分のみ」「抜粋」という表現を使用禁止**
- **ユーザーから指摘を受けた場合、即座に完全版を提供**

### 2. 修正作業の進め方
- **段階的実行を徹底**
  - ステップ1: 実行内容を明示
  - 確認: 結果を待つ
  - ステップ2: 結果に基づいて次の指示
- 複数の修正を一度に提示しない
- 各ステップの結果確認後に次へ進む

### 3. SQL実行時のルール
- 「現状確認」SQLを提示する場合は、必ず以下を明記：
  - 期待される出力結果の例
  - 出力結果から何を判断すべきか
  - 正常/異常の判断基準

### 4. エラー対応
- エラーが発生したら、まず原因特定のための確認作業を指示
- 推測で複数の修正案を提示しない
- 確実な原因が判明してから修正を提示

### 5. ルールの更新管理
- **やり取りの中で新しいルールが必要になった場合：**
  - 即座にこのファイルへの追記を提案する
  - 「CLAUDE_INSTRUCTIONS.mdに以下を追加してください」という形で明示
  - ユーザーから指摘された改善点は必ずルール化する
- **ルール違反があった場合：**
  - ユーザーから指摘を受けたら、該当するルールを追加/修正
  - 同じミスを繰り返さないよう、具体的なルールとして文書化

### 6. プロジェクトファイルの確認
- **提案する際、必ず以下のファイルを確認すること：**
  - `complete_source.txt`: 全ソースコード
  - `database_schema.txt`: Supabaseのテーブル構造とRLSポリシー
  - その他プロジェクトに添付されているファイル
- **確認せずに提案することは禁止**
- 修正提案時は、他のファイルへの影響も考慮する

### 7. 文字数制限への対応
- チャットの文字数制限が近づいてきた場合、ユーザーに通知する
- 次のチャットへの引き継ぎ情報として以下を提案：
  - `CLAUDE_INSTRUCTIONS.md`の更新案
  - `PROJECT_STATUS.md`の更新案
  - 未完了のタスクリスト
- 引き継ぎ時は、現在の作業状態を明確に記載

## プロジェクト情報

### 技術スタック
- Next.js 14 (App Router)
- TypeScript
- Supabase (PostgreSQL + Auth)
- Tailwind CSS

### アプリケーション概要
- 草野球チーム管理Webアプリケーション「草野球 is Good」
- チーム管理、試合管理、スコア入力機能を提供

### データベース構造
- `teams`: チーム情報
- `team_members`: チームメンバー（外部キー: auth.users）
- `team_join_requests`: チーム参加申請
- `games`: 試合情報
- `game_scores`: イニングごとのスコア
- `user_profiles`: ユーザープロフィール

### 重要な実装済み機能
- 認証: Supabase Auth (Google/GitHub OAuth)
- チーム作成時にオーナーを自動的にteam_membersに追加（トリガー）
- チーム参加申請の承認/却下機能
- RLSポリシーによるアクセス制御

## 現在の課題・注意点
- team_membersのuser_idはauth.usersを参照（user_profilesではない）
- RLSポリシーは可能な限りシンプルに保つ（無限再帰を避ける）
- 文字数制限があるため、効率的な対応を心がける

## このファイルの使い方
新しいチャットセッションを開始する際：
「プロジェクトのCLAUDE_INSTRUCTIONS.mdファイルを確認して、その指示に従って対応してください」
と伝える。
